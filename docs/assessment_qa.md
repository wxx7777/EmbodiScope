# EmbodiScope v2.3 答辩高频问题

## 1. 这个项目与普通日志可视化工具有什么本质区别？

Rerun、Foxglove 等工具主要解决多模态数据呈现。EmbodiScope 进一步负责质量判定、同步偏移估计、失败首因定位、保守清洗、局部恢复计划和配对恢复验证，并且可以把结果反向导出到 Rerun。核心差异是“证据闭环与评测协议”，不是新的绘图库。

## 2. 你的原创部分到底是什么？

原创边界包括统一诊断 Schema、数据适配语义映射、故障注入与全量状态采集、同步诊断和风险评分、保守清洗策略、FaultBench、RepairBench、连续状态到任务谓词的落地、失败传播解释、RecoveryLab 配对协议、RecoveryBench 安全分离统计以及 Web 回放状态机。物理引擎、文件解析和基础渲染由开源组件承担。

## 3. RecoveryLab 如何证明恢复具有因果效果？

Failure 和 Recovered 固定环境、机器人、控制器、随机种子、故障参数、相机和仿真预算，只改变首因之后是否执行恢复干预。任务成功增量、谓词恢复、算子完成、路径代价和安全指标来自这组受控差异。

## 4. 为什么任务恢复了，完整 Episode 仍然不安全？

碰撞在恢复触发前已经发生。恢复动作可以降低后续风险并完成任务，但不能抹去历史安全违规。因此项目分别报告 `task_recovery`、`episode_safety` 和 `post_intervention_safety`，三者互不替代。

## 5. 为什么 collision / seed 10 被算作恢复失败？

Failure 和 Recovered 最终都成功，恢复干预没有带来任务成功上的反事实增益。即使谓词和后续安全门成立，也不能声称“恢复使任务从失败变成成功”，因此严格判定为失败。

## 6. 为什么排除 seed 8，会不会是挑数据？

准入规则在分析前定义：两个变体必须包含受控故障签名。seed 8 在故障注入前第 5 帧已经由环境报告成功，没有形成待恢复故障，因此不属于实验问题。排除原因、原始帧数和结果均写入 `excluded_seeds`，并顺延候选补足样本，没有删除不利结果。

## 7. 88.9% 是否具有统计显著性？

九组配对只能证明框架在当前确定性任务中的可运行性，不能证明通用恢复策略。项目报告 Wilson 95% 区间 `56.5%-98.0%`，主动暴露小样本不确定性，并把多任务、多控制器和真机验证列为下一阶段。

## 8. 为什么现在只有 PickCube-v1？

优先完成端到端实验协议，包括故障签名、在线触发、配对完整性、任务与安全分离和统计导出。单任务让变量可控，但限制外部效度。下一步应扩展到 PushCube、StackCube、PegInsertion 等具有不同接触和任务结构的场景。

## 9. 为什么没有直接训练一个恢复策略？

当前工作的目标是工具和模块开发，先建立可信的故障证据、恢复接口和评测协议。没有可靠协议时，学习策略可能只优化最终成功率并掩盖安全违规。RecoveryBench 可以作为后续学习型恢复策略的统一对照基准。

## 10. 任务谓词是不是手工规则？

当前谓词是可解释的物理规则，输出 `true / false / unknown`、证据和置信度。它们用于建立可审计接口，不声称是通用语义模型。未来可加入视觉语言模型或学习型状态估计，但仍应保留物理安全谓词作为硬约束。

## 11. 阈值能否迁移到其他机器人？

鲁棒统计阈值按轨迹计算，物理安全门和工作空间由 Profile 管理。迁移到新机器人时，需要依据 URDF、控制器限位和传感器标定生成新 Profile，不能直接复用 Panda 的力和速度限制。

## 12. 如何证明清洗没有把异常抹平？

RepairBench 单独检查正常轨迹过度修复率、误隔离率、风险隔离召回和物理测量保持率。碰撞、滑脱、卡滞等事实保留原值，只通过 `quality_valid` 控制训练准入。当前正常轨迹误修复和误隔离均为 0%。

## 13. 如何证明仿真和回放不是预置动画？

每次执行都会生成新的作业编号、HDF5、MP4、replay JSON 和缩略图。接触力来自 PhysX 接触查询；一键诊断后由独立适配器重新读取 HDF5。可以现场比较新作业时间戳、文件路径和导出的诊断结果。

## 14. 为什么 Windows 演示使用 CPU 渲染？

当前驱动组合在 SAPIEN 离屏 GPU 渲染时存在访问冲突。交付配置使用 `physx_cpu + sapien_cpu`，优先保证现场稳定；接触、刚体状态和 RGB 仍来自真实 SAPIEN/PhysX 执行，并非录制动画。

## 15. 项目最重要的不足是什么？

科学上是单任务、单控制器和小样本；工程上是尚未接入真机持续数据流和人工标注闭环；方法上是规则谓词还不能覆盖开放类别视觉失败。项目当前结论严格限制在确定性仿真和已实现协议内。

## 16. 下一步最有价值的扩展是什么？

优先构建多任务 RecoveryBench：覆盖不同接触模式、对象几何和长时任务，再加入行为树、MoveIt Task Constructor 和学习型恢复策略作为基线。随后接入真机回放与人工复核，评估仿真到真实的恢复迁移。
