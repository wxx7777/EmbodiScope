# EmbodiScope 技术报告

## 1. 问题背景

具身智能实验中的一次任务执行，往往同时产生控制指令、关节状态、末端位姿、RGB/深度图像、力觉和任务标签。不同信号由不同设备和进程产生，常见问题包括传感器掉帧、时钟漂移、动作突跳、碰撞、抓取滑脱和日志缺失。

现有流程通常依赖多个独立脚本：先检查 CSV，再打开视频，再用绘图工具对齐时间轴，最后人工记录失败原因。这种流程耗时、难复现，也难以在大规模训练数据入库前执行统一质量门禁。

EmbodiScope 的目标是把一次人工排查转化为可重复的自动流程：**输入一份轨迹日志，输出质量评分、异常证据、失败根因、训练准入清单与可追溯清洗产物。**

## 2. 目标用户与使用流程

目标用户包括机器人算法研究者、数据采集团队、模型评测人员和真机部署工程师。

```text
CSV / LeRobot / ManiSkill / ROS bag / MCAP / 在线 PhysX 仿真
      ↓
适配器识别、Schema 映射与 Episode 切分
      ↓
完整性 ─ 时序 ─ 运动 ─ 同步 ─ 安全检测
      ↓
风险约束评分与证据融合
      ↓
连续状态谓词 → 技能算子图 → 首因定位与恢复规划
      ↓
批量质量门禁 / 保守清洗与风险隔离
      ↓
SAPIEN 视频 + Panda 数字孪生 / Rerun / CSV / JSON / Markdown 报告
```

## 3. 系统架构

系统按职责分层：

| 层级 | 模块 | 职责 |
|---|---|---|
| 仿真层 | `embodiscope.simulation` | 闭环控制 Panda、注入故障、采集 RGB/接触力/状态并导出可回放轨迹 |
| 数据集层 | `DatasetLibrary` | 管理白名单数据源、来源许可证、规模统计和安全切换 |
| 适配层 | `embodiscope.adapters` | 识别 CSV、LeRobot、ManiSkill、ROS/MCAP，展开向量字段并对齐异步话题 |
| 数据层 | `validate_dataframe` | 统一 Schema 校验、类型转换、Episode 切分 |
| 分析层 | `analyze_episode` | 执行八类检测器与五维评分 |
| 评测层 | `FaultBench`、`AnalysisProfile` | 多种子故障注入、固定阈值对照、指标汇总与物理参数配置 |
| 治理层 | `dataset_overview`、`embodiscope.repair` | 跨 Episode 审计、可逆数值校正、风险隔离、来源与产物哈希 |
| 推理层 | `embodiscope.task_reasoning` | 连续信号落地为任务谓词，检查算子约束并生成局部恢复计划 |
| 恢复实验层 | `embodiscope.recovery_lab`、`embodiscope.recovery_benchmark` | 在同条件故障对照下执行局部恢复，拆分任务/安全结论并做多场景多种子统计 |
| 解释层 | 根因排序、`build_markdown_report` | 将统计结果转换为问题、证据和建议 |
| 交互层 | HTTP API、Three.js、Rerun、CLI | 单条排查、三维回放、批量分析和结果导出 |

Web 服务使用 Python 标准库实现，时序图表使用 Canvas，空间轨迹使用本地化 Three.js ES Module。运行时不依赖外部 CDN，也不需要 Node 构建流程。

### 3.1 开源生态集成

- **Apache Arrow / PyArrow**：按列读取 LeRobot Parquet；目录模式只选择 episode 数据文件，避免混入 meta Parquet，并读取版本、视觉 feature 与 MP4 元数据。
- **h5py / ManiSkill schema**：读取 trajectory group 下的 actions、qpos、TCP 位姿、目标位置、力觉和任务结果。
- **rosbags**：统一读取 ROS1 bag、ROS2 SQLite 和 ROS2 MCAP，反序列化标准消息类型。
- **MCAP**：作为可流式读取、跨机器人生态的日志容器；项目生成真实 ROS2 MCAP 演示数据进行回归测试。
- **Three.js**：在浏览器中绘制轨迹、TCP 游标、起点与异常事件位置，提供播放、拖拽、缩放和三种标准视角。
- **Rerun SDK**：将统一 Schema 重新组织为三维实体、标量序列和事件日志，生成可离线交付的 `.rrd` 文件。
- **OpenCV**：当 ROS 日志包含 `Image` 或 `CompressedImage` 时，计算低分辨率帧差作为 `camera_motion`。
- **ManiSkill 3 / SAPIEN 3 / PhysX**：提供 `PickCube-v1` 环境、Panda 机器人、真实接触求解与 RGB 相机；平台实现数值雅可比反馈控制、故障注入和轨迹导出。
- **Hugging Face LeRobot PushT**：固定修订引入 206 条视觉操作 Episode；读取 v3 Parquet、任务元数据和合并视频偏移，保留 MIT 许可证与论文引用。

开源库负责成熟的文件格式、消息反序列化和图形基础设施；项目自身负责安全 ZIP 解包、字段语义映射、异步时间对齐、统一 Schema、异常诊断、三维回放状态机、Rerun 实体布局和解释输出。

### 3.2 多来源数据集目录

`data/dataset_catalog.json` 只登记项目根目录内的可信路径，服务端按 ID 加载，拒绝客户端提交任意文件路径。目录当前汇总 4 个来源、214 条 Episode 和 30,450 个采样点。开源 PushT 文件使用固定修订与 SHA-256 校验，避免上游更新导致答辩结果漂移。

对于 PushT，`observation.state` 是二维像素坐标而非机器人关节。适配器依据元数据映射为 `state_1/state_2`，并保留动作、奖励、结束标记和任务文本；任务成功标签缺失时以“未标注”展示，不生成虚假的失败根因。

## 4. 关键技术

### 4.1 鲁棒离群检测

对关节速度和接触力使用中位数绝对偏差：

```text
threshold = median(x) + k × 1.4826 × median(|x - median(x)|)
```

与均值和标准差相比，MAD 不容易被少数极端点拉高，更适合在本身包含异常的机器人日志上计算阈值。同时设置物理下限，避免静止轨迹产生过低阈值。

### 4.2 多模态同步估计

系统由末端位置求速度模长，将其与 `camera_motion` 标准化后，在 ±500 ms 窗口内枚举时间偏移并计算 Pearson 相关系数。最大相关位置即为估计偏移；正值表示相机信号晚于机器人状态。

这种方法不要求额外同步标记，只利用“机器人运动应当引起画面变化”这一跨模态共有事件。

### 4.3 任务阶段感知的卡滞检测

单纯检测低速会把等待和复位误判为卡滞。EmbodiScope 先排除 `idle/reset` 阶段，再寻找末端速度低于 0.004 m/s 且持续 1.2 秒以上的片段，以保留任务语义。

### 4.4 风险约束评分

基础得分由五个维度加权得到：完整性 22%、时序 20%、运动 20%、同步 18%、安全 20%。在此基础上加入两条约束：

- 每个严重问题增加固定风险惩罚，并将总分上限限制为 79。
- 任一维度低于 40 时，总分上限限制为 69。

这样可以避免“其余信号都很完整，所以发生碰撞的轨迹仍有 90 分”的不合理结果。

### 4.5 空间与时间统一回放

Three.js 工作台将原始 `(ee_x, ee_y, ee_z)` 转换为以 Z 为竖直轴的浏览器场景。完整轨迹、已播放轨迹、当前 TCP 和异常事件使用不同几何体表达。播放状态机按真实 timestamp 推进，而不是按固定帧序号推进，因此不规则采样数据仍保持正确速度。

Rerun 导出使用相同时间基准写入 `episode_time` timeline。浏览器和 `.rrd` 因此共享同一事件位置与诊断结论，避免出现“图上看到的时刻”和“导出记录时刻”不一致。

### 4.6 真实仿真与故障注入

闭环运行器使用 `pd_joint_delta_pos` 控制模式。每个控制周期通过有限差分计算 Panda TCP 对七个关节的数值雅可比，并使用阻尼最小二乘求解关节增量。该方案不依赖额外 Pinocchio 安装，能够在考核环境中稳定复现。

系统在物理步进期间记录关节、策略动作、实际执行动作、TCP、物体、目标、全部连杆位置、归一化夹爪开度、`is_grasped`、技能阶段、奖励、成功标记和各连杆最大接触力。基准控制器依次执行 approach、reach、grasp 和 transport，固定种子 7 下约在第 65 帧完成抓取到目标，而不是只进行一次未完成的抓取尝试。

仿真目录扩展为十场景矩阵：碰撞直接改变执行器目标产生真实接触力；抓取滑脱在物理抓取建立后施加一次 60 N 横向外力；夹爪失效保留策略闭合意图但改写实际执行动作；卡滞冻结控制增量 1.5 秒；动态目标扰动在接近阶段移动方块并检验闭环重定位；视觉侧包含 200 ms 延迟、连续 6 帧丢失、10 帧遮挡和碰撞+延迟+丢帧复合压力测试。这些注入会进入 HDF5、MP4 和 replay JSON，并再次经过正式适配器和诊断器。

Windows + RTX 4050 环境中，SAPIEN GPU 离屏渲染存在驱动级访问冲突，因此交付配置固定为 `physx_cpu + sapien_cpu`。这牺牲部分吞吐量，但 40 步仿真和视频编码约 1 秒完成，并保留真实 PhysX 接触与 SAPIEN RGB 画面，现场稳定性更高。

### 4.7 诊断 Profile 与统计评测

诊断阈值不再散落在界面或脚本中。`AnalysisProfile` 统一记录关节速度 MAD 系数与物理下限、卡滞速度和持续时间、同步偏移与置信度、接触力阈值、丢帧时长、滑脱条件和 XYZ 工作空间。当前提供通用操作机器人和 Franka Panda 两个 Profile；每次 Episode 分析与 Benchmark 导出都附带完整配置。

FaultBench 从同一条 600 行正常多模态轨迹出发，对视觉延迟、接触力峰值、关节突跳、时间缺口、连续丢帧、执行卡滞和抓取滑脱分别注入轻微、中等、严重三档强度。每个随机种子还包含一条只加入测量噪声的正常轨迹。正式诊断器与保守固定阈值规则读取完全相同的样本，并统一计算宏平均 Precision、Recall、F1、Exact Match、正常误报率、同步偏移 MAE、时间定位误差和 p50/p95 延迟。

关节运动速度同时修正为 `diff(q) / diff(t)`，单位为 `rad/s`，避免相同运动在 10 Hz 和 100 Hz 日志中得到不同卡滞结论。对应回归测试同时验证两种采样率。

### 4.8 批量审计与可追溯清洗

`dataset_overview` 对每条 Episode 复用正式诊断器，不在前端重复实现规则。结果同时保留问题代码、五维得分和缺失率，并聚合等级分布、分数区间、故障频次、维度均值、训练就绪数量和最差 Episode 排名。审计 JSON/CSV 附带源文件或数据集目录的确定性 SHA-256。

清洗模块将动作分为三类：`correction` 只用于有严格局部证据的短缺口、孤立突跳和视觉时间偏移；`segmentation` 在异常时间间隔后建立新片段，不补造样本；`quarantine` 用于碰撞、滑脱、卡滞、无效视觉帧、越界和无法安全插值的缺失片段。物理故障的原始数值始终保留，训练侧只读取 `quality_valid=true` 的样本。

每个数值校正都创建 `__original` 备份列；每行记录 `source_row / repair_actions / repair_reason / segment_id`。manifest 保存处理动作、影响时间范围、保留率、修复前后问题集合、完整 Profile、源数据 SHA-256 与清洗 CSV SHA-256，因此能够回答“哪一行被谁按什么规则修改”以及“导出文件是否发生变化”。

### 4.9 RepairBench 与整套数据集交付

只验证诊断准确率不能证明清洗可靠。RepairBench 从同一条 600 行基准轨迹出发，对短缺口、孤立关节突跳、视觉时间偏移、时间戳缺口和真实接触力峰值分别施加轻微、中等、严重三档损坏；每个种子还包含一条正常噪声轨迹。评测不只检查动作代码是否出现，还分别计算重建 RMSE、同步残差、分段召回、风险隔离召回、物理测量保持率、正常轨迹过度修复率和误隔离率。

批量清洗管理器在后台逐 Episode 调用同一清洗器，支持进度、取消和状态恢复。完成后合并为带 `quality_valid` 训练门的 Zstandard Parquet，同时生成逐 Episode CSV、数据血缘 manifest 和 ZIP。服务端对作业编号与产物类型做白名单校验，下载响应禁用缓存，避免浏览器或代理复用旧训练数据包。

### 4.10 连续谓词、任务图与恢复算子

任务推理层把 pick-and-place 拆为 `ApproachObject → ReachPregrasp → SecureGrasp → TransportObject → ReleaseAtGoal` 五个算子。每个算子显式声明前置条件和效果，例如运输必须满足 `object_attached / collision_free / motion_progress`，放置必须满足 `at_goal / object_attached / collision_free`。

谓词不是从阶段名称直接硬编码为真：`object_attached` 联合夹爪开度、目标距离和滑脱诊断，`collision_free` 使用力觉与工作空间门限，`motion_progress` 使用位姿速度与卡滞诊断，`observation_fresh` 使用时间戳、有效帧和同步诊断。证据不足时状态保持 `unknown` 并降低落地覆盖率。`at_goal` 在缺少目标位姿时允许由 `transport→place` 转换做中置信度推断，界面明确展示该证据等级。

失败推理优先选择有时间定位的最早诊断事件，将其映射到失效谓词和当前技能，再生成失败传播链与恢复算子。恢复计划按安全、控制、感知、规划、验证排序，只恢复缺失谓词。例如滑脱触发停止运输、张开夹爪、重定位、重规划预抓取、闭合并验证附着，而不是从头重做整个任务。该结果是规划建议，不声明机器人已执行或恢复成功。

### 4.11 RecoveryLab 配对恢复验证

任务推理回答“应该如何恢复”，RecoveryLab 进一步回答“这次恢复是否真实有效”。每个实验固定环境、随机种子、故障、相机和仿真步数，顺序运行 Failure 与 Recovered 两组。两组都保留同一故障；Recovered 只在首因出现后执行局部算子，因此不会把无故障 `nominal` 轨迹包装成恢复成功。

恢复层当前覆盖碰撞、夹爪执行失效和抓取滑脱。触发器逐帧读取真实接触力、夹爪命令/开度和 `is_grasped`，分别识别力阈值越界、连续命令响应失配和抓取状态丢失；恢复从检测后的下一控制步启动，不使用固定步骤表。仿真写入 `predicate-violated / recovery-start / predicate-restored / recovery-success` 事件和 `success_trace`，使触发原因、恢复延迟、谓词恢复和算子完成状态可由轨迹验证。

结果同时计算任务成功差值、恢复延迟、恢复后力峰、共同观测窗口 TCP 路径开销和算子完成率。六项质量门要求配对指纹一致、Failure 预期失败、Recovered 成功、首因谓词恢复、算子有序完成和恢复阶段力峰受控。两组 MP4、力觉曲线、36 N 安全线和事件轨道使用同一时间游标，评委可以从触发证据直接跳转到对应物理画面。

### 4.12 Safety-aware RecoveryBench

单次恢复通过不能回答鲁棒性，也容易产生“最终成功等于过程安全”的错误表述。v2.3 将每次结果拆为 `task_recovery`、`episode_safety`、`post_intervention_safety` 三个独立 verdict：任务恢复检查因果配对、失败对照、谓词恢复与有序算子；整段安全审计完整 Recovered Episode；干预后安全只审计在线恢复触发后的轨迹。碰撞故障主动制造 36 N 以上冲击，因此即使后续撤退、重规划并成功放置，正确结论仍是“任务恢复、整段不安全、干预后安全”。

RecoveryBench 默认跨三个故障，从起始 seed 开始准入三个有效随机种子，执行 9 组 Failure/Recovered 配对。准入条件只检查实验是否真正包含受控故障及可比初态，不读取恢复成功结果；故障注入前环境已终止的候选会公开记录并顺延补样。批量模式关闭 MP4 录制但保留 HDF5/replay，并在全局物理仿真锁下串行运行。总体与逐场景结果报告恢复率、配对完整率、在线触发覆盖率、两种安全率、恢复延迟、路径代价和算子完成率；任务恢复率使用 Wilson score 95% 区间，避免把少量样本的 100% 点估计表述为确定泛化。

## 5. 原创点

1. **从异常点到诊断证据链**：每个检测结果包含级别、类别、时间范围、统计证据和修复建议，而非只输出布尔值。
2. **数据质量与任务失败联合分析**：同一界面将采集问题、控制问题和任务问题关联起来，区分“日志坏了”和“机器人做错了”。
3. **无标记多模态偏移估计**：利用末端运动与视觉运动的自然相关性，自动给出可执行的毫秒级校正量。
4. **风险约束评分机制**：严重安全问题不能被其他高分维度平均掉，更符合真机部署与训练数据门禁需求。
5. **完全自包含的可复现演示**：确定性生成六种实验场景，评委无需安装仿真器或连接机器人即可验证全部能力。
6. **插件式跨生态数据层**：同一碰撞轨迹分别编码为 CSV、LeRobot Parquet、ManiSkill HDF5 和 ROS2 MCAP，不同适配器输出一致诊断结果。
7. **诊断事件的空间化表达**：异常不再只显示在表格和时间轴上，还映射到机器人末端轨迹的真实三维位置。
8. **双层可视化交付**：浏览器工作台适合现场演示，Rerun `.rrd` 适合工程团队继续深挖多模态记录。
9. **可验证的仿真诊断闭环**：故障注入、物理结果、视频帧、结构化轨迹与诊断结论可以沿同一时间轴相互校验。
10. **数字孪生同步回放**：真实 SAPIEN RGB 与 Three.js Panda 共享帧索引，既能查看真实画面，又能从任意视角检查机器人和物体状态。
11. **Profile 化物理先验**：统计阈值、机器人安全边界和任务规则共同进入可追溯配置，而不是隐藏在代码常量中。
12. **可复现统计验证**：用多随机种子、多强度故障和固定阈值对照量化算法收益，而不是只展示单个成功案例。
13. **诊断驱动的数据治理**：同一分析结果同时驱动批量质量门禁和单条清洗，不产生前后端规则漂移。
14. **物理事实与数据修复分离**：只校正可验证的数据问题，真实碰撞、滑脱和卡滞只隔离并保留原值。
15. **双哈希可追溯产物**：源数据和清洗 CSV 分别计算 SHA-256，配合原值备份和逐行动作形成可审计血缘。
16. **修复结果的独立质量门**：RepairBench 将“成功执行清洗动作”与“重建正确、同步正确、物理量未被篡改、正常数据未被误修”分开验收。
17. **从单条预览到训练数据包**：同一治理策略可扩展到整套数据集，并以 Parquet、逐集摘要、manifest 和 ZIP 形成可直接进入训练流水线的交付物。
18. **从信号异常到任务恢复**：将连续日志落到可审计的前置条件、技能效果和安全不变量，定位最早失效谓词，并给出只恢复缺失条件的局部算子序列。
19. **跨域仿真压力矩阵**：同一成功基准控制器上分别注入接触、执行器、任务扰动、视觉和复合故障，同时记录策略意图、实际执行和物理抓取状态，避免只堆叠离线数据扰动。
20. **同条件配对恢复验证**：故障组与恢复组共享环境、种子和故障，只改变首因后的局部干预，并以谓词、力觉、路径和任务结果共同验收。
21. **恢复与安全解耦统计**：RecoveryBench 将最终任务恢复、完整历史安全和干预后安全分开，并以多种子矩阵和 Wilson 区间约束泛化表述。

## 6. 效果验证

数据集库包含 30,450 个采样点、214 个 Episode，并提供 CSV、LeRobot Parquet + MP4、ManiSkill HDF5 与 ROS2 MCAP 四种编码。自动测试验证：

FaultBench 1.0 使用 8 个固定随机种子，形成 `176` 条独立评测轨迹：8 条正常轨迹，以及 `7 类故障 × 3 档强度 × 8 个种子`。结果如下：

| 指标 | EmbodiScope | 固定阈值 Baseline | 差值 |
|---|---:|---:|---:|
| Macro Precision | 97.1% | 96.1% | +1.0 pp |
| Macro Recall | 100.0% | 79.2% | +20.8 pp |
| Macro F1 | 98.4% | 86.1% | +12.3 pp |
| Exact Match | 96.6% | 77.8% | +18.8 pp |
| 正常轨迹误报率 | 0.0% | 0.0% | 0.0 pp |

同步偏移 MAE 为 `0.0 ms`，可定位故障的时间中位误差为 `0.0 ms`。当前设备单条 600 行轨迹分析 p95 低于 `50 ms`；延迟受设备负载影响，因此 JSON 同时保留本次实测值。评测可通过 `python scripts/run_benchmark.py --seeds 8` 重新生成，完整结果位于 `output/benchmark/faultbench-v1.0.json`。

全部 `6/6` 质量门通过。评测可通过 `python scripts/run_repair_benchmark.py --seeds 4` 重新生成，完整结果位于 `output/benchmark/repairbench-v1.0.json`。

RecoveryLab 在 `seed=7 / horizon=140` 的真实物理回归中得到：

| 场景 | Failure | Recovered | 恢复延迟 | 恢复后力峰 | 算子 |
|---|---|---|---:|---:|---:|
| 碰撞 | `success=false` | `success=true` | 约 3.6 s | 约 28.0 N | 5/5 |
| 夹爪失效 | `success=false` | `success=true` | 约 2.9 s | 约 27.9 N | 5/5 |
| 抓取滑脱 | `success=false` | `success=true` | 约 3.9 s | 约 35.3 N | 6/6 |

三类实验均由在线物理谓词触发，恢复首因谓词并通过六项质量门。碰撞 Recovered Episode 的完整力峰约 `290 N`，所以三分判定为 `任务恢复=PASS / 整段安全=FAIL / 干预后安全=PASS`；这比单一 PASS 更准确。该结果证明具体恢复干预在受控场景中有效，不被解释为跨任务策略泛化率。

RecoveryBench 真实批测从 `seed=7` 起准入 3 个有效种子，最终接受 `7、9、10`；`seed=8` 因环境在故障注入前第 5 帧已成功而公开排除。统计结果为：

| 指标 | 结果 |
|---|---:|
| 有效配对 | 9 |
| 严格任务恢复率 | 88.9%（8/9） |
| Wilson 95% CI | 56.5%-98.0% |
| 完整 Episode 安全率 | 66.7% |
| 干预后安全率 | 100.0% |
| 在线触发覆盖率 | 100.0% |
| 配对完整率 | 100.0% |
| 恢复延迟 p95 | 3.93 s |

夹爪失效与抓取滑脱均为 `3/3` 严格恢复；碰撞为 `2/3`。唯一失败项 `collision / seed=10` 中，Failure 在约 `144 N` 碰撞后仍完成任务，Recovered 也成功。恢复谓词、5/5 算子和干预后安全均成立，但干预没有产生任务成功的反事实增益，因此不能计为严格任务恢复。该失败揭示了只看最终 `success=true` 会高估恢复策略效果。

整套 6 Episode 演示数据的批量清洗处理 `3,600` 行，修改 `814` 行、隔离 `219` 行、保留 `3,381` 行，保留率为 `93.92%`。最终 ZIP 包含 `cleaned.parquet / episode_summary.csv / manifest.json`，可通过 `python scripts/run_batch_repair.py` 独立复现。

| 场景 | 真实注入 | 系统输出 |
|---|---|---|
| 正常 | 无异常 | 无严重告警，100.0 分 |
| 同步错误 | 相机延迟 160 ms | 估计偏移 160 ms |
| 碰撞 | 约 50 N 力峰 | 定位 7.18–7.28 s |
| 卡滞 | 2.7 s 位姿冻结 | 定位 5.32–8.00 s |
| 控制异常 | 两次关节突跳、240 ms 缺口 | 检出 4 个突跳边沿与 1 个时序缺口 |
| 抓取失败 | 闭合夹爪下目标距离增大 | 判定抓取滑脱，并发现 25 个缺失值 |

对于这组数据，手工逐列绘图和记录通常需要数分钟；CLI 分析在当前设备上低于 1 秒，且输出格式一致、可批量复现。

适配器验证不使用伪造返回值：测试通过 `rosbags.Writer` 写入 ROS2 消息，通过 h5py 写入 ManiSkill 层级轨迹，通过 PyArrow 写入 LeRobot v3 目录，再由正式适配器读取。真实仿真回归进一步启动 ManiSkill：正常场景 `success=true` 且约 28 N 接触不触发碰撞；卡滞、夹爪失效和滑脱分别输出 `ROBOT_STUCK`、`GRIPPER_RESPONSE_FAILURE` 和 `GRASP_SLIP`；动态目标扰动后仍能闭环完成任务；复合场景同时产生 `FORCE_SPIKE`、`SENSOR_DESYNC` 和 `FRAME_DROP`。

浏览器验证覆盖桌面与 `390 × 844` 移动视口，无页面横向溢出；移动端 Benchmark、审计排名、任务谓词证据表、清洗证据表和 RecoveryBench 矩阵只在自身容器内滚动。Hugging Face PushT 的 206 条 Episode、25,650 个采样点实测约 3.0 秒载入、1.35 秒完成审计聚合。仿真 MP4 为 `320 × 240`；PushT 合并视频为 `96 × 96`、总时长 `2565 s`，按 Episode 1 可准确跳到 `16.1–27.9 s`，两者均达到 `readyState=4` 并支持 Range 请求。当前 `58/58` 项自动测试通过，其中三类 RecoveryLab 均验证在线触发、Failure 失败、Recovered 成功、配对指纹、谓词恢复、有序算子完成和独立安全结论；RecoveryBench 另验证 Wilson 区间、聚合口径与历史结果恢复。

## 7. 不足与扩展

- 支持 LeRobot 视频帧解码和远程 Hugging Face Dataset 流式读取。
- 扩展更多 ManiSkill 环境、机器人控制器和并行批量仿真队列。
- 将 Profile 扩展到更多机器人，并根据 URDF 自动生成关节限位与工作空间约束。
- FaultBench 当前以可控轨迹扰动和真实 ManiSkill 回归为主，后续需引入大规模真机人工标注失败集验证跨平台泛化。
- RepairBench 当前以可控损坏和正常轨迹保护为主，后续需在大规模真机清洗集上增加双人复核和一致性评估。
- RecoveryBench 当前覆盖三个确定性物理故障和有限随机种子，后续需增加更多任务、视觉闭环恢复基线和真机干预评测。
- 使用无监督聚类归纳跨 Episode 的新失败模式。
- 接入实验平台，在数据入库或模型发布前自动执行质量门禁。
- 将任务模板扩展为可配置 PDDL/行为树，并接入 MoveIt 或 PDDLStream 生成可执行恢复计划。
