# EmbodiScope v2.3 考核答辩与现场演示脚本

本脚本对应 `docs/EmbodiScope_Assessment_Deck_v2.3.pptx`。总汇报控制在 15 分钟，其中 PPT 讲解约 11 分 30 秒，现场演示约 3 分钟，结尾保留 30 秒。

## 15 分钟答辩讲稿

### 0:00-0:50｜第 1 页：项目主张

“我选择的是课题 2，开发一个面向具身智能实验的工具与模块。项目名称是 EmbodiScope。它解决的不是单纯的数据可视化，而是把具身实验失败变成可复现、可诊断、可恢复验证的证据。

当前系统能够运行真实 ManiSkill、SAPIEN 和 PhysX 仿真，统一采集视频、机器人状态、动作、接触力和任务阶段；在诊断之后进一步生成任务谓词和局部恢复计划，并通过 Failure / Recovered 配对实验验证恢复是否真的有效。”

### 0:50-1:50｜第 2 页：问题背景

“具身实验失败至少包含五层信息：控制器发出了什么动作，机器人和视觉发生了什么变化，是否出现接触或任务失败，系统采取了什么恢复干预，以及这个干预能否跨随机种子稳定生效。

传统工作流的断点是：日志、视频和仿真分散；恢复建议经常没有真实执行；任务最终成功又容易被直接解释成安全。我的判断是，只有保留同一个故障，固定环境和初始条件，并比较不恢复和恢复两个变体，才能验证恢复的因果收益。”

### 1:50-2:50｜第 3 页：系统设计

“系统形成四段闭环。第一段是真实仿真，运行 PickCube-v1 和 Panda 控制器并产生真实 PhysX 接触。第二段是统一证据，视频、动作、状态、接触力和任务阶段使用同一时间轴。第三段把连续状态落到任务谓词，定位最早失效的前置条件或安全不变量。第四段进行配对验证，分别评估任务成功、安全性和路径代价。

同一个 Episode ID 贯穿 HDF5、MP4、replay JSON、诊断结果和恢复结果，因此每一项结论都能回到原始物理帧。”

### 2:50-4:00｜第 4 页：三个原创层次

“第一个原创层次是物理故障证据链。同一个注入事件会写入视频、状态、力觉和诊断时间轴，正式适配器还会重新读取刚生成的 HDF5，而不是直接使用仿真器预置标签。

第二个层次是保守数据治理。短缺口和孤立数值突跳可以修复，但碰撞、滑脱等物理事实只隔离，不会被篡改成正常数据。RepairBench 中正常轨迹的误修复率和误隔离率都是 0%。

第三个层次是反事实恢复验证。Failure 和 Recovered 固定环境、种子、故障、相机和仿真预算，只改变首因之后是否执行局部恢复；任务恢复和安全性作为两个独立结论报告。”

### 4:00-4:55｜第 5 页：RecoveryLab 真实运行

“这里是 RecoveryLab 的真实配对回放。左侧保留碰撞故障但不执行恢复，右侧保留完全相同的故障并执行五个局部恢复算子。碰撞案例 seed 7 中，Failure 最终失败，Recovered 最终成功，五个算子全部完成，恢复延迟为 3.60 秒。

需要强调的是，恢复组仍然保留已经发生的碰撞。因此正确结论不是简单的成功，而是任务恢复、完整 Episode 不安全、干预后安全。”

### 4:55-5:45｜第 6 页：同步证据回放

“系统不是预录动画。真实仿真会生成新的作业编号、HDF5、MP4、回放 JSON 和缩略图。视频、Three.js Panda、接触力、动作和诊断事件由同一个帧索引驱动。

点击 1.90 秒的碰撞事件，可以同时定位视频画面、机器人位姿、136.8 N 的瞬时接触力和 fault_collision 阶段。整段真实 PhysX 峰值约为 157 N。”

### 5:45-6:45｜第 7 页：任务与运动推理

“日志异常本身还不能告诉机器人应该从哪里恢复。因此系统建立五个 grounded operator，把连续轨迹映射到 observation_fresh、object_attached、collision_free 等谓词。

EP-003 中最早失效的是 TransportObject 阶段的 collision_free。系统不从头重做任务，而是生成 EmergencyStop、Retreat、Reobserve、Replan 和 ForceGuardedRetry 五个局部算子。当前六个演示 Episode 中四个失败可局部恢复，谓词落地覆盖率为 99.2%。”

### 6:45-7:55｜第 8 页：RecoveryLab 验收协议

“RecoveryLab 的关键不是执行了动作，而是验收动作是否改变了物理结果。碰撞案例中任务结果从 false 变为 true，首因谓词被恢复，五个算子全部有执行证据。

完整 Episode 的峰值仍为 290.2 N，因为注入碰撞历史没有被删除；从 recovery-start 开始的峰值是 27.9 N，低于 36 N 安全线。恢复增加了 0.1818 米路径，因此系统同时报告收益和代价。”

### 7:55-9:10｜第 9 页：RecoveryBench 效果验证

“为了避免只展示一个成功案例，我运行了 RecoveryBench。它覆盖碰撞、夹爪执行器失效和抓取滑脱三个场景，以及种子 7、9、10，共九组有效配对。

严格任务恢复为 8/9，也就是 88.9%，Wilson 95% 置信区间为 56.5% 到 98.0%。完整 Episode 安全率只有 66.7%，但干预后安全率、在线触发覆盖和配对完整率均为 100%。

唯一严格失败是 collision / seed 10：Failure 和 Recovered 都成功，因此恢复干预没有产生任务成功上的反事实增益，不能把它计为恢复成功。seed 8 在故障注入前第 5 帧已经成功，未出现受控故障签名，因此在分析前公开排除并顺延补样。”

### 9:10-10:15｜第 10 页：开源与原创边界

“项目使用成熟开源基础设施：ManiSkill、SAPIEN 和 PhysX 提供物理环境，PyArrow、h5py 和 rosbags 提供文件读取，Three.js 和 Rerun 提供可视化基础。

项目原创部分是故障注入和全量采集、统一 Schema、异步流对齐、诊断与保守清洗算法、grounded predicate、局部恢复计划、RecoveryLab 配对协议、RecoveryBench 统计协议以及整套 Web 状态机和产物管理。BehaviorTree.CPP 和 MoveIt Task Constructor 是方法依据，没有复制它们的规划器实现。”

### 10:15-11:15｜第 11 页：工程完成度

“工程上，平台支持 CSV、LeRobot、ManiSkill 和 ROS2/MCAP 四类数据，内置 214 条 Episode。FaultBench、RepairBench 和 RecoveryBench 分别验证诊断、数据治理和恢复干预。

当前 58 项自动化测试全部通过。交付物包括代码、README、v2.3 PPT、15 分钟讲稿、3 分钟演示路径、备用视频和完整 JSON 证据。启动脚本会检查端口、健康接口、关键结果文件和仿真运行时，降低现场环境风险。”

### 11:15-11:55｜第 12 页：限制与扩展

“当前已经证明的是：在确定性仿真中，失败能够被复现、诊断并通过配对实验验证局部恢复。当前边界也很明确：RecoveryBench 仍然只有 PickCube-v1、九组有效配对和单一 Panda 控制器，不能替代多任务泛化或真机安全认证。

下一阶段将扩展到多 ManiSkill 任务、多机器人、动态场景和真机人工标注失败集，并加入学习型恢复策略作为对照。”

## 3 分钟现场演示路径

演示前运行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\assessment_start.ps1
```

不要现场重新运行完整 RecoveryBench。直接加载已完成的正式结果，只执行短仿真或单组 RecoveryLab。

### 0:00-0:35｜真实仿真与同步回放

1. 打开“仿真回放”。
2. 选择“碰撞注入”，40 步，点击“启动真实仿真”。
3. 作业完成后点击碰撞事件，指出视频、Panda、接触力和任务阶段同步跳转。
4. 说明新作业同时生成 HDF5、MP4 和 replay JSON。

### 0:35-1:05｜独立诊断

1. 点击“一键送入诊断”。
2. 指出数据源是刚生成的 `sim-*.h5`。
3. 展示 `FORCE_SPIKE`、时间范围、统计证据和建议。
4. 强调诊断器重新读取物理产物，不依赖预置结论。

### 1:05-1:40｜任务推理

1. 点击顶部“任务推理”。
2. 选择 EP-003。
3. 指出 `collision_free=false`、失败传播链和五个局部恢复算子。
4. 用一句话说明：这里生成的是可解释恢复计划，还不是恢复成功证明。

### 1:40-2:35｜RecoveryLab 与 RecoveryBench

1. 点击“恢复实验”。
2. 直接展示已恢复的 collision / seed 7 配对结果。
3. 指出三个独立 verdict：`RECOVERED / UNSAFE / SAFE`。
4. 播放 Failure / Recovered 同步视频，点击 recovery-start 事件。
5. 向上展示 RecoveryBench：任务恢复 89%、整段安全 67%、干预后安全 100%、Wilson 区间和 seed 矩阵。

### 2:35-3:00｜结论

“这条演示链证明了四件事：故障是真实执行的，诊断来自独立读取，恢复计划来自首因谓词，恢复效果由配对物理实验和多种子统计验证。系统不会用任务成功掩盖已经发生的安全违规。”

## 现场故障切换

- Web 服务异常：播放 `docs/EmbodiScope_Assessment_Demo_v2.3.mp4`，继续按相同讲稿解释。
- ManiSkill 启动失败：跳过新仿真，使用历史 `recovery-20260721-205407-7bd9f9` 和正式 RecoveryBench。
- 视频无法播放：直接展示回放 JSON、接触力曲线和同步截图。
- 时间不足：省略重新仿真，从“任务推理”开始，RecoveryLab 和 RecoveryBench 必须保留。
